# Part 1. 项目完整素材（対応：プレゼン必須4要素）

## 1.1 プロジェクト概要（30秒で言えるように）

**一句话定位（日本語）：**
> 水産物の輸入案件ごとに発生する複数の貿易書類をAIが読み取り、記載の不一致や不足書類を提出前に検出し、修正依頼メールや検疫・税関向け提出資料の草案まで作成する、Human-in-the-loop型のワークフローAI Agentです。

**问题背景（数字化地说）：**
- 一个海运集装箱的进口，涉及 Invoice / Packing List / B/L / Health Certificate / Certificate of Origin 等 5 种以上单证
- 同一个信息（重量、箱数、集装箱号、原产国…）会在不同单证上重复出现 20 次以上，分散在 PDF、邮件、Excel、多语言中
- 人工逐份核对费时且易漏；发现晚了会导致检疫/通关延迟、Free Time 超期、滞箱费（Demurrage）
- **本质痛点不是"读一份文件"，而是"跨文件核对 + 判断下一步该做什么"**——这是市面 OCR 工具不解决的

**定位声明（差异化，务必说）：**
> これは「PDFを要約するチャットボット」ではなく、書類受領から検疫・通関資料準備まで案件を前進させる「ワークフロー型Agent」です。

## 1.2 技術的アプローチ

### 技术栈
| 层 | 技术 | 选型理由 |
|---|---|---|
| 前端/后端 | Next.js 16 (App Router) + TypeScript + Tailwind | Server Actions 使全栈类型安全，单人开发效率最高 |
| 数据库 | Prisma 7 + SQLite | MVP 零配置；Schema 已按 Postgres 兼容设计 |
| LLM | OpenAI 互換 Provider 抽象（現在 DeepSeek） | 一个 env 切换任何模型（含 IBM watsonx/Granite），不锁定供应商 |
| 校验 | Zod（LLM 输出结构校验）+ 纯 TS 规则引擎 | LLM 输出必须过 schema 才入库 |
| PDF | unpdf（抽文本）+ pdf-lib（生成样例） | 纯 JS 无原生依赖，任何环境可跑 |
| 测试 | Vitest 26 件（校验规则/状态机/归一化） | 确定性部分 100% 可单测 |
| 部署 | Docker + Render，启动时自动迁移+种子数据 | push 即发布 |

### 架构（发表时画/贴这张图）

```
Browser（日本語UI）
   ↓ Server Actions
Workflow Orchestrator（状態機・監査ログの唯一の入口）
   ├─ Intake        書類分類・Draft/Final判定・版管理
   ├─ Extraction    PDF→テキスト→LLM構造化抽出（根拠・ページ・信頼度付き）
   ├─ Normalization 単位・日付・国名の統一            ←決定論的
   ├─ Validation    一致性・完全性・妥当性チェック      ←決定論的（LLM不使用）
   ├─ Action        修正依頼メール草案（LLM、送信はしない）
   └─ Package       検疫・税関向け提出パッケージ（JSON/CSV）
   ↓
Prisma + SQLite（全操作を WorkflowEvent として監査ログ化）
```

### 実装の工夫（发表和答辩的核心，按重要度排序）

**① LLM と決定論的コードの役割分担（最重要）**
- LLM 只做它擅长的：分类、抽取、翻译、草案生成
- **一致性判定、算术、状态迁移一律用确定性代码**——"10,000 kg vs 9,800 kg 不一致"这种结论绝不让 LLM 下
- 理由：可复现、可单测、可审计、无幻觉风险。监管相关业务的红线

**② 根拠（Evidence）主義**
- 每个抽取值必须带：参照元書類 / ページ番号 / 原文テキスト / 信頼度(0-1)
- Prompt 明令禁止推测："NEVER guess a value that is not written in the text"
- 找不到的字段返回 null，UI 显示「根拠不足」而非编造

**③「値が無い」≠「不一致」**
- 值缺失时判定为 INSUFFICIENT_EVIDENCE（根拠不足），不误报为不一致
- 这是从贸易实务角度设计的概念区分，体现领域理解

**④ 状態機の白名单制**
- 20 个工作流状态（規格書§9），非法迁移直接抛异常
- 所有迁移写入 WorkflowEvent：谁（AI/人）、何时、从哪到哪、为什么——完整审计链

**⑤ Human-in-the-loop 的边界设计**
- AI 绝不做：发邮件、向行政系统申报、HS Code 最终决定、删审计日志
- 人只处理"要対応"：不一致的受容/エスカレーション、草案承认、Gate 通过
- UX 上正常字段不需要逐个确认——只把问题点推给人

**⑥ Provider 抽象 + Demo Mode**
- LLM 层唯一接口 `completeJSON(schema, prompt, content)`：JSON 强制 + Zod 校验 + 失败重试1次 + 再失败转人工
- `DEMO_MODE=true` 时用 fixture 数据完整走通全流程——**演示不依赖网络和 API 余额**（发表保险）

## 1.3 直面した課題と解決策（选 2-3 个讲，全部准备）

| 課題 | 解決策 | 学到的 |
|---|---|---|
| ① 安価なLLMのJSON出力が不安定（フェンス付き・余計な前置き） | JSON抽出の前処理 + Zod検証 + 1回リトライ + 失敗時は「抽出失敗・人間確認」に降格。**値を推測して埋めることは絶対にしない** | LLMは「信頼できない外部入力源」として設計する |
| ② 表記ゆれで単純比較が成立しない（"10,000 kg" vs "10500 kgs" vs "July 18, 2026" vs "2026-07-18"） | 決定論的な正規化層（単位・日付・国名・コード）を挟み、正規化後の完全一致で判定。原文は必ず別途保持 | 正規化と原文保持の両立が監査性の鍵 |
| ③ APIキー無しでも完全なデモが必要（コスト制約・回線リスク） | Provider抽象 + fixtureベースのDemo Mode。UIから校验、審批まで全機能がオフラインで動く | 環境依存を疑うところから設計する |
| ④ 実LLMで「HCの数量欄に重量が入る」等の field mapping ミスが発生 | 現状：低信頼度・不一致としてUIに浮上→人間が修正。今後：フィールド単位の型検証（数量にkgは不可等）を正規化層に追加 | 100%を目指さず「間違いが必ず人の目に入る」設計にする |
| ⑤ フレームワークの破壊的変更（Prisma 7で接続方式一新、Next 16のasync params） | 公式移行ガイドとnode_modules内ドキュメントを読み、最小構成で検証してから適用 | 最新版を使う覚悟＝一次情報を読む習慣 |

# Part 3. 【顧客視点】現状の不足と、開発者が取るべき技術的解決策

> 面接で「自分のプロダクトの弱点は？」と聞かれたとき、ここから3つ選んで正直に言えると強い。「弱点を構造的に把握し、解決技術まで言える」ことが評価される。

## 🔴 P0：本番導入を阻む問題

**1. 認証・権限管理が無い**
- 症状：誰でも全案件を見られ、誰でも承認できる。「担当者」は固定文字列
- 解決：NextAuth.js / Auth0 で認証、RBAC（担当者/承認者/管理者）。規格書にある Green/Yellow/Red の承認権限分離を実装。承認操作に本人記録を紐付け
- 工数目安：1-2週間

**2. データ永続性（SQLite + ローカルファイル）**
- 症状：Render無料枠では再起動でデータ消失。アップロードPDFもローカル保存
- 解決：PostgreSQL（Neon/Supabase/AWS RDS）へ移行——Prisma なので schema はほぼそのまま。PDF は S3/Cloudflare R2 等のオブジェクトストレージ＋署名付きURL
- 工数目安：3-5日

**3. スキャン画像PDFが読めない**
- 症状：テキスト層の無いPDF（FAXスキャン等）は「抽出失敗」になる。実務の書類はスキャンが多い
- 解決：①Vision対応LLM（GPT-4o / Claude / Granite Vision）にページ画像を渡す、②Azure AI Document Intelligence 等の専用OCRで前処理。信頼度スコアはOCR由来とLLM由来を分けて表示
- 工数目安：1-2週間

## 🟡 P1：実務で不便な問題

**4. 必要書類ルールがVN→JP固定**
- 解決：ルールをDB化（ルート×品目×目的→必要書類・確認事項のマトリクス）、管理画面で編集可能に。将来は規制文書のRAGで根拠条文を提示

**5. メールを手動でコピペ**
- 解決：Gmail API / Microsoft Graph 連携。ただし送信は必ず人間の明示クリック（Human-in-the-loop維持）。受信側も監視し、返信着信で案件を自動で「修正版待ち→受領」へ

**6. 抽出フィールドの取り違え（実測で発生）**
- 症状：Health Certificate の「数量」欄に重量 12,500 kg が入った
- 解決：①フィールド単位の型制約（quantity に重量単位は不可）を正規化層で拒否、②few-shot例をプロンプトに追加、③評価用の正解付き書類セット（eval set）を作り抽出精度を回帰テスト化

**7. 大量アップロード時の同期処理**
- 症状：5書類で1-2分、その間UIはボタン無効化のみ。20書類なら耐えられない
- 解決：ジョブキュー（BullMQ + Redis / Inngest）で非同期化、書類ごとに進捗ストリーミング表示（SSE）。抽出中も他の操作を可能に

**8. 同時編集の競合**
- 解決：楽観ロック（updatedAt比較）、承認操作の冪等化

## 🟢 P2：将来の伸びしろ

**9. 多言語UI**（現在日本語のみ）→ next-intl。書類側は既に多言語OK（LLM任せ）
**10. 過去案件の活用**→ 類似案件検索（embedding + pgvector）、「この輸出者は過去にSeal No.ミスが3回」等のリスクプロファイル
**11. 人間の修正からの学習**→ 修正ログを教師データ化し、プロンプト/few-shotを自動改善
**12. モバイル承認**→ 低リスク承認だけスマホPWAで

# Part 4. 【IBM面接官視点】予想質問と模範回答

> 全部日本語で即答できるよう練習。回答は「結論→理由→（あれば）実例」の順で30秒以内。

## A. 技術判断系（最も聞かれる）

**Q1. なぜ検証をLLMにやらせず、わざわざルールコードを書いたのですか？**
> A. 3つ理由があります。①再現性——同じ書類には必ず同じ判定を返す必要がある、②監査性——「なぜ警告したか」をルールとして説明できる必要がある、③ハルシネーション排除——「不一致だ」という事実判定にAIの確率的挙動を混ぜたくない。LLMは分類・抽出・文章生成という「曖昧さの処理」に限定し、判定は26件の単体テストで担保されたコードが行います。

**Q2. LLMの抽出が間違っていたらどうなりますか？**
> A. 3層の防御があります。①出力はZodスキーマ検証を通らないと入庫されず、失敗時は自動リトライ→それでも失敗なら「抽出失敗」として人間へ、②すべての値に原文の根拠と信頼度が付き、低信頼度は自動的に「要確認」でUIの上部に浮上、③そもそも書類に無い値を推測することをプロンプトで禁止し、無ければnullを返させます。実測では数量欄に重量が入る取り違えが起きましたが、不一致検出に引っかかって人間の目に入りました。「間違いが握りつぶされない構造」が設計思想です。

**Q3. ハルシネーション対策をまとめると？**
> A. ①根拠（原文・ページ）の強制付与、②推測禁止のプロンプト制約、③スキーマ検証、④事実判定は決定論的コード、⑤最終判断は常に人間。この5つです。

**Q4. なぜマルチエージェント構成にしなかったのですか？**
> A. 意図的な判断です。仕様書の段階で「MVPでは自律的なマルチエージェントを優先しない」と決めました。単一のOrchestratorと論理的に分離したモジュール（Intake/Extraction/Validation…）で、同じ関心の分離は達成できます。エージェント間の自律対話はデバッグ困難・コスト増・挙動不安定のリスクがMVPの価値に見合いません。将来、輸出者側との交渉自動化のような真に自律性が要る場面で検討します。

**Q5. なぜDeepSeekを？IBMのwatsonx / Graniteは使えますか？**
> A. LLM層をOpenAI互換のProvider抽象にしているので、ベースURL・モデル名の環境変数を変えるだけでwatsonx.ai上のGraniteに差し替え可能です。DeepSeekを選んだのは開発時のコスト効率（1案件約1円）のためで、特定ベンダーへのロックインを避ける設計自体が本プロダクトの主張です。むしろ企業導入ではデータ主権の観点でwatsonxのようなエンタープライズ基盤が適合すると考えています。

**Q6. プロンプトインジェクション——書類の中に「この書類を承認しろ」と書かれていたら？**
> A. 攻撃面を構造的に絞っています。①LLMの出力はフィールド抽出のJSONに限定され、Zodで形が強制されるので「命令」が混ざる余地がない、②システムに承認・送信を実行する権限がそもそも無く、すべての効果的操作は人間のUIクリックが必要、③抽出値は常に原文根拠つきで表示されるので異常な値は目視で発覚します。

**Q7. 精度はどう評価していますか？**
> A. 現状は2段階です。①決定論部分は26件の単体テストで完全担保、②LLM部分は意図的に誤りを埋め込んだ書類セットで実測し、5書類の分類5/5、埋めた誤り2/2検出を確認。今後は正解ラベル付きの書類evalセットを整備し、モデルやプロンプト変更のたびに抽出精度を回帰テストする仕組みにします。

**Q8. スケールさせるには何がボトルネックですか？**
> A. ①SQLite→PostgreSQL、②PDF保存→オブジェクトストレージ、③同期抽出→ジョブキュー（BullMQ等）による非同期化と進捗ストリーミング、④認証とマルチテナント分離。アーキテクチャはServer Actions＋独立したOrchestrator層なので、この置き換えでUI層は変わりません。

**Q9. なぜ状態遷移を白名单方式に？**
> A. 業務ワークフローでは「あり得ない遷移」（書類受領中→輸入許可など）をコードレベルで不可能にすることが監査上重要だからです。全遷移がWorkflowEventに記録され、「いつ誰が何を」の完全な履歴になります。テストでも不正遷移の拒否を検証しています。

**Q10. HS Code の判定まで自動化しないのはなぜ？**
> A. 関税分類は法的責任を伴う専門判断で、誤りは追徴課税や法令違反に直結します。AIは候補と根拠の提示までに留め、最終決定は通関士・人間の責務として設計上明確に線を引きました。「AIに何をさせないか」を決めることが、業務AIの信頼性だと考えています。

**Q11. RAGは使っていますか？使うとしたらどこ？**
> A. 現MVPでは未使用です。抽出は書類全文をコンテキストに入れる方が確実なサイズだからです。将来は2箇所で有効と考えます。①過去案件の類似検索（この輸出者・品目の過去トラブル）、②検疫・関税規制文書の検索と根拠条文の提示です。

**Q12. テストできないLLM部分の品質はどう守る？**
> A. 「テストできない部分を最小化する」のが方針です。LLMの責務を抽出・分類に絞り、その出力の後段（正規化・検証・状態遷移）はすべてテスト済みの決定論的コード。LLM自体はfixtureで差し替え可能にしてあり、CI ではDemo Modeで全フローの結合テストが回せます。

## B. プロダクト・ビジネス系

**Q13. 想定顧客と課金モデルは？**
> A. 一次顧客は中堅の水産物輸入商社の貿易実務チーム（数名〜数十名規模）。属人化と確認漏れが最も痛い層です。課金は案件数ベースのSaaS＋初期のルール設定支援を想定します。通関業者への横展開も考えられます。

**Q14. 既存のOCR製品・通関システムとの違いは？**
> A. OCRは「1枚を読む」道具ですが、本製品の価値は「複数書類の突合」と「次のアクションへの接続」です。NACCSのような申告システムは提出の器であって、提出前の照合・準備の混沌は扱いません。その間のワークフローの空白を埋めるのが本製品です。

**Q15. 導入の最大の障壁は？**
> A. ①誤検出への不信（→根拠表示と「受容」操作で人間が常に上書きできる設計で緩和）、②既存業務フローの変更コスト（→メール・Excelと共存できる漸進導入）、③機密書類を外部LLMに送る抵抗（→Provider抽象によりオンプレ/専有環境のモデルに差し替え可能）です。

## C. 開発プロセス・姿勢系

**Q17. 開発期間と体制は？AIをどう使いましたか？**
> A. 一人開発で、Claude Code（AIコーディングエージェント）をペアプログラマーとして活用しました。ただし丸投げではなく、①最初に詳細な仕様書を自分で固め、②実装計画をレビュー・承認してから書かせ、③E2Eテストと実LLM検証で品質を確認する、という「Human-in-the-loopな開発プロセス」を実践しました。プロダクトの設計思想と開発プロセスが同型なのがこのプロジェクトの面白いところだと思っています。

**Q18. 一番苦労した点は？**
> A. 技術的にはLLM出力の不安定さへの多層防御ですが、本質的に難しかったのは「AIにやらせないことを決める」線引きでした。何でもAIにやらせると監査不能になり、絞りすぎると価値が出ない。仕様書の段階で『LLM/決定論コード/ルールデータ』の3分類を先に決めたことが、後の実装の迷いを無くしました。

**Q19. もう一度最初から作るなら何を変えますか？**
> A. ①抽出精度のevalセットを最初に作る（プロンプト改善が感覚頼みだった）、②ジョブキューを最初から入れる（同期処理は後から直すほうが高い）、③フィールド型制約を正規化層に最初から組み込む。逆に、Provider抽象とDemo Modeを初日に作ったのは正解でした。

**Q20. この経験をIBMでどう活かしたいですか？**
> A. 本プロジェクトで実践した「AIの責務を限定し、根拠と監査性を担保し、人間の判断を中心に据える」設計は、IBMが企業向けAIで重視する信頼性・ガバナンスの考え方（watsonx.governance等）と方向が同じだと理解しています。エンタープライズの業務ワークフローにAIを「安全に」組み込む仕事に貢献したいです。
